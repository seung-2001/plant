Plant App 데이터베이스 구조
=========================

1. 사용자(User) 데이터베이스
---------------------------
컬렉션명: users
{
    _id: ObjectId,
    name: String,          // 사용자 이름
    email: String,         // 이메일 (로그인용)
    password: String,      // 비밀번호 (암호화 저장)
    profileImage: String,  // 프로필 이미지 URL
    volunteerHours: Number,// 봉사 시간
    createdAt: Date,      // 계정 생성일
    updatedAt: Date       // 정보 수정일
}

2. 게시물(Post) 데이터베이스
---------------------------
컬렉션명: posts
{
    _id: ObjectId,
    content: String,       // 게시물 내용
    author: ObjectId,      // 작성자 ID (users 컬렉션 참조)
    likes: [ObjectId],     // 좋아요한 사용자 ID 배열
    comments: [{          // 댓글 배열
        content: String,   // 댓글 내용
        author: ObjectId,  // 댓글 작성자 ID
        createdAt: Date    // 댓글 작성일
    }],
    createdAt: Date,      // 게시물 작성일
    updatedAt: Date       // 게시물 수정일
}

3. 봉사활동(Volunteer) 데이터베이스
--------------------------------
컬렉션명: volunteers
{
    _id: ObjectId,
    title: String,        // 봉사활동 제목
    description: String,  // 봉사활동 설명
    date: Date,          // 봉사활동 날짜
    time: String,        // 봉사활동 시간
    location: String,    // 봉사활동 장소
    requirements: [String], // 필요한 준비물 배열
    maxParticipants: Number, // 최대 참가자 수
    currentParticipants: Number, // 현재 참가자 수
    organizer: ObjectId,  // 주최자 ID (users 컬렉션 참조)
    participants: [ObjectId], // 참가자 ID 배열
    status: String,      // 봉사활동 상태 (open/closed/completed)
    createdAt: Date,     // 생성일
    updatedAt: Date      // 수정일
}

데이터베이스 관계
---------------
1. 게시물(Post) -> 사용자(User)
   - author 필드가 users 컬렉션의 _id를 참조
   - likes 배열이 users 컬렉션의 _id들을 참조
   - comments.author가 users 컬렉션의 _id를 참조

2. 봉사활동(Volunteer) -> 사용자(User)
   - organizer 필드가 users 컬렉션의 _id를 참조
   - participants 배열이 users 컬렉션의 _id들을 참조

3. 사용자(User) -> 봉사활동(Volunteer)
   - volunteerHours 필드가 참여한 봉사활동의 시간을 합산하여 저장

인덱스
-----
1. users 컬렉션
   - email: unique (이메일 중복 방지)

2. posts 컬렉션
   - author: index (작성자별 게시물 조회 최적화)
   - createdAt: index (시간순 정렬 최적화)

3. volunteers 컬렉션
   - date: index (날짜별 조회 최적화)
   - status: index (상태별 조회 최적화)
   - organizer: index (주최자별 조회 최적화)
